Single person teams

The Tech Stack of a One-Man SaaS@ panelbear: * Company: PanelBear. Powerful, privacy-friendly analytics to grow your website.. * blog post * hn discussion

Languages: Backend: python + django

Frontend: * typescript * react Libraries: * NextJs: for landing pages, documentation, blog * Bootstrap4: because it's widely documented

Databases: * Clickhouse: Because it's incredibly fast for analytical queries. It's also open source. * PostgreSQL: For RDBMS. Sane defaults, battle-tested and deeply integrated with django. * Redis: Because it's rock solid and has solid documentation. Used for Caching, rate limiting, task queue and key-value store with ttl.

Key takeaways: python+django+postgres is what makes the one man thing possible. No need to reinvent wheels. Quickly and reliably get a sane pre-cooked framework.

Deployment philosophy: * Treat servers as cattle instead of pets. Things are not changed by sshing into the server. A template is stored on github which is used to spin up servers with a single command.

Tech used for deployment: * Terraform: Things like EKS clusters, S3 buckets, roles, and RDS instances are declared in the terraform manifests. The state is synced to an encrypted S3 bucket in case something happens to the dev laptop. * Docker: everything is built as docker images. Even stateful components (clickhouse, redis) are packaged and shipped as docker containers * Kubernetes: To timplify operational aspects. Though, won't blindly recommend it. Author relies on managed offerings for less headache. * Github Actions: Replacement for CircleCi as it removes yet another service which needs access to his repositories and deployment secrets. However CircleCi is still recommended.

Infrastructure: Digital ocean 5$ -> Digital ocean Managed kubernetes -> Linode -> AWS DO had uptime problems

  • Aws: predictable and lots of managed services
  • Cloudflare: DDoS protection, serving dns, offloading edge caching of static assets
  • Let's Encrypt: Free SSL
  • NameCheap: Registrar of choice

Kubernetes coponents: * ingress-nginx: sits behind the nlb * cert-manager * external-dns * prometheus-operator: exposes dashboards via Grafana * flux

Cli tools: * kubectl * stern * htop * cURL: Issue HTTP requests locally * HTTPie: for JSON APIs * hey: load testing http endpoints. Gives nice latency distribution summary.

Monitoring * Prometheus: To store time series data for monitoring * Grafana * Sentry: Exception minotoring and aggregation. Notifications for unhandled errors with additional metadata * Loki: Log aggregation. Bundled with ptometheus-operator

Email: * Fastmail: Business email * Sendgrid: Transactionl email(verigication, login alerts, pwd reset etc). Note: ip addresses are blicked by some email providers. Author is considering moving to Postmark.

Development: * Github * Pycharm * VS Code * Poetry: Python packaging and dependency management * Yarn * Invoked: To wrap all codebase tasks in invoked commands. For example inv build will prepare static assets, package the frontend/backend distribution, and build the docker image. This way, can locally run the same commands that the dev would run on CI, in case it's needed.

Other: * PanelBear Analytics: Yeah, he uses his own website to track his own website * Healthchecks: Notifies via email/whatsapp when a scheduled job doesn't run (Bootstrapped saas) * Trello * Figma: Replaced sketch

Hn suggestions:

  • mvanga

    • having gone down this path a year back, just use Heroku for your backend.
    • Maybe Netlify if you want to separate your frontend and backend. Heroku costs more, but there's a reason for it.
    • If you're even remotely making money from your product, and are alone in running the show, it's a no brainer to drop a couple hundred dollars a month to eliminate 99% of your ops headaches: the documentation, the add-on's, the fantastic UI, decent reliability, and ease of operations are totally worth it. Literally the worst thing one can say about Heroku is the cost, and possibly, the inability to build certain complex architectures (which most single-person SaaS services aren't).

  • joshmn, personal website: Serial one-man SaaS builder here with a pair of soft-landings and one big hit:

    • I would agree with others who have opined that this is overkill. You couldn't pay me to use Terraform or Kubernetes for my SaaS unless it was a much larger team.
    • My stack usually consists of this: A framework I am comfortable with that removes a lot of boilerplate (happens to be the one I know best in the language I know best), the database I know best, the memory store I know best, Turbolinks for load times, a theme from ThemeForest (with the commercial license), self-hosted analytics, Amplitude, and Heroku.
    • The author mentions "lessons learned" regarding Clickhouse and Kubernetes. I'm not sure what their original goal was when starting their SaaS. Usually mine is financially motivated more than learning things.
    • What I'm really saying is that just go with what you know best. Having to scale is a good problem to have. Pre-optimization is the root of all evil — asking "is x the right stack for y?" will always be yes when you ask in the x subreddit.
    • Stop thinking and build.

  • zomgwat

    • As a one-man team operating a 10+ year old SaaS product, I’ve done two things that have helped keep things sustainable. The frontend continues to be server generated. PJAX style partial page updates is (mostly) dynamic enough. The maintenance burden of operating a JavaScript frontend is too high to justify for a 1-2 person team.
    • The second thing I’ve done is avoid containers. VMs work fine for many types of applications. That’s especially true if provisioning and configuration is mostly automated. A product like Google Cloud Run looks interesting for a few low-volume backend services that I operate. Containers are a good fit there because it simplifies operations rather than increases complexity. The core infrastructure will remain VMs for the foreseeable future.
    • I began doing Ruby and Ruby on Rails (RoR) development in 2006. Ruby had been around for awhile but the growing popularity of RoR was driving a lot of changes in the ecosystem in those years. It was a fun time but I was working for a large company that allowed me the time to keep up with everything. I see today’s frontend and container ecosystems in the light of that experience. I’m now more content to wait it out and let the dust settle as much as I enjoy learning new technology.
